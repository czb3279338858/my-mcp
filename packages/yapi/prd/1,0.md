# PRD：YApi 接口智能搜索 MCP Server

## 一、产品定位
一个基于 MCP 协议的服务器，让 AI 能够智能搜索 YApi 平台中的接口。

核心能力：
- 根据请求参数和/或响应字段精准搜索接口（工具一）
- 根据关键词模糊搜索接口（工具二，直接调用 YApi API）
- 兼容 YApi 现有数据结构，不依赖数据治理规范

## 二、MCP 工具定义

### 工具一：search_yapi_interface
用途：精准搜索。根据请求参数和/或响应字段，查找能完成特定数据转换的接口。

输入参数：
- param (string, 可选): 请求参数名或参数注释中的关键词，如 orgId、组织ID
- response (string, 可选): 响应字段名或字段注释中的关键词，如 orgName、组织名称
- limit (integer, 可选): 返回数量，默认 3，最大 10

注：param 和 response 至少提供一个。两者都提供时，为 AND 关系。

搜索范围（仅字段级）：
- 请求参数名
- 请求参数注释
- 请求参数是否必填（用于返回，不用于匹配）
- 响应字段名（从 res_body JSON 解析）
- 响应字段注释（从 res_body 注释解析）
- 响应字段类型（用于返回，不用于匹配）

匹配逻辑：
- 大小写不敏感，子串匹配
- 优先匹配字段名（exact），其次匹配注释（comment）
- 返回结果按匹配质量排序（exact > comment）

返回格式：
```json
{
  "status": "ready",
  "data": [
    {
      "name": "获取组织详情",
      "path": "/api/org/info",
      "method": "GET",
      "description": "根据组织ID查询组织详情",
      "project_name": "组织管理",
      "request_params": [...],
      "response_fields": [...],
      "match_type": "exact",
      "matched_on": {"param": "name", "response": "comment"}
    }
  ]
}
```

缓存状态说明：
- ready: 缓存就绪，data 包含搜索结果
- loading: 缓存正在加载中，data 为空
- error: 缓存加载失败，data 为空

loading 状态返回示例：
```json
{
  "status": "loading",
  "message": "接口数据正在加载中，预计 10-30 秒后完成，请稍后重试",
  "estimated_time": 15,
  "data": []
}
```

### 工具二：search_yapi
用途：通用搜索。直接调用 YApi 的 /api/interface/search API 进行实时查询。

输入参数：
- keyword (string, 必填): 搜索关键词，如 组织、登录、/api/org
- limit (integer, 可选): 返回数量，默认 5，最大 20

返回格式：
```json
{
  "status": "success",
  "data": [...]
}
```

失败返回：
```json
{
  "status": "error",
  "message": "YApi API 调用失败：{错误原因}"
}
```

## 三、缓存设计与状态管理

### 3.1 缓存用途
- 工具一依赖缓存进行字段级精准搜索
- 工具二不依赖缓存，实时调用 YApi API
- 工具二的每次调用会触发后台缓存更新检查

### 3.2 缓存状态机
服务启动 -> 读取本地缓存文件（如果存在）-> status = "loading"（立即返回，不等待）-> 后台异步拉取 YApi 最新数据 -> 拉取完成，写入缓存 -> status = "ready"

关键原则：
- 服务启动后立即将状态设为 loading
- 更新期间不使用旧缓存响应查询
- 查询请求统一返回 loading 状态
- 更新完成后自动切换到 ready

### 3.3 搜索流程
工具一：
search_yapi_interface -> 检查缓存状态 -> ready: 使用缓存数据搜索返回结果；loading: 返回 loading 状态；error: 返回 error 状态

工具二：
search_yapi -> 调用 YApi /api/interface/search 返回结果 -> 同时异步检查缓存是否过期，过期则触发后台全量拉取更新缓存

### 3.4 缓存配置
| 环境变量 | 默认值 | 说明 |
| --- | --- | --- |
| YAPI_CACHE_PATH | C:\yapi-mcp\cache.json | 缓存文件绝对路径 |
| YAPI_CACHE_TTL | 1440 | 缓存有效期（分钟）。0 代表每次请求都触发更新 |
| YAPI_REQUEST_TIMEOUT | 30000 | HTTP 请求超时（毫秒） |

### 3.5 缓存数据结构
```json
{
  "version": "1.0",
  "lastUpdated": "2026-06-08T10:30:00Z",
  "projects": [
    {"_id": 123, "name": "组织管理", "basepath": "/api/org"}
  ],
  "interfaces": [
    {
      "_id": 456,
      "title": "获取组织详情",
      "path": "/info",
      "method": "GET",
      "desc": "根据组织ID查询组织详情",
      "project_id": 123,
      "project_name": "组织管理",
      "fullPath": "/api/org/info",
      "request_params": [...],
      "response_fields": [...]
    }
  ]
}
```

### 3.6 缓存过期判断
def is_expired(self) -> bool:
    if self.ttl == 0: return True
    if not self.data or not self.data.get('lastUpdated'): return True
    last_updated = datetime.fromisoformat(self.data['lastUpdated'])
    return datetime.now() - last_updated > timedelta(minutes=self.ttl)

## 四、YApi 接口调用

### 4.1 认证方式
POST /api/user/login
Content-Type: application/json
{"email": "{{YAPI_EMAIL}}", "password": "{{YAPI_PASSWORD}}"}

### 4.2 全量数据拉取流程（工具一缓存）
1. 认证 -> POST /api/user/login -> 获取 Cookie
2. 获取项目列表 -> GET /api/project/list
3. 遍历项目获取接口列表 -> GET /api/interface/list?project_id={id}
4. 遍历接口获取详情 -> GET /api/interface/get?id={interface_id}
5. 数据预处理与缓存（解析 res_body、合并请求参数、计算完整路径、写入缓存）

### 4.3 实时搜索流程（工具二）
1. 确保已认证（Cookie 有效）
2. 调用搜索 API -> GET /api/interface/search?keyword={keyword}&limit={limit}
3. 返回结果

### 4.4 响应字段解析
从 res_body JSON 字符串中解析：
1. 解析 JSON 字符串为对象
2. 提取字段名作为 name
3. 提取字段类型作为 type
4. 提取注释（// 后的内容）作为 description

## 六、AI 使用策略

### 工具一描述
工具名：search_yapi_interface
用途：根据请求参数或响应字段精准搜索接口
示例：param="orgId", response="orgName"

### 工具二描述
工具名：search_yapi
用途：根据关键词模糊搜索接口（实时查询）
示例：keyword="组织", keyword="/api/org"

### AI 调用策略
1. 优先使用 search_yapi_interface
2. 如果返回 loading，告诉用户稍后重试
3. 如果返回 ready 但 data 为空，调用 search_yapi 兜底
4. 用户只提供业务关键词时，直接使用 search_yapi
